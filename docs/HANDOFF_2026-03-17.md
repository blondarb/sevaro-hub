# Handoff — 2026-03-17

## Session 1: What's New System (Phase 1)

Built Phase 1 of the "What's New" cross-app update notification system for Sevaro Hub.

### What Was Built

**AWS Infrastructure:**
- DynamoDB table `sevaro-whats-new` (PK: `appId`, SK: `timestamp`, pay-per-request)
- Lambda `sevaro-whats-new-api` (Node 22, `aws-jwt-verify` for Cognito JWT validation)
- API Gateway `5168ofhh8k` (HTTP API with CORS, 4 routes)
- IAM inline policy `dynamodb-whats-new` on `sevaro-lambda-basic` role

**API Endpoints:**
- `GET /whats-new?appId=...&since=...` — public, returns app-specific + "all" entries, strips `createdBy`
- `GET /whats-new/all` — admin-only, returns all entries (used by admin UI)
- `POST /whats-new` — admin-only, creates entry with category/link validation
- `DELETE /whats-new` — admin-only, removes entry by composite key

**Hub Frontend:**
- `src/lib/whats-new-api.ts` — typed API client (getUpdates, getAllUpdates, createUpdate, deleteUpdate)
- `src/components/WhatsNewBadge.tsx` — dropdown badge with unseen count, 1-hour sessionStorage cache, localStorage last-seen tracking
- `src/app/admin/whats-new/page.tsx` — admin CRUD page with stats, app filter, create form, delete
- `src/components/NavBar.tsx` — added WhatsNewBadge + admin "What's New" link
- `lambda/sevaro-whats-new-api/index.mjs` — Lambda source with `aws-jwt-verify`

### Cross-Model Review

Codex CLI (GPT) review completed. All Critical and High findings addressed:
1. JWT signature verification (Critical — fixed)
2. Auth on `/all` endpoint (High — fixed)
3. `createdBy` leak on public responses (High — fixed)
4. JSON parse error handling (Medium — fixed)
5. Link URL scheme validation (Medium — fixed)
6. localStorage render loop (Low — fixed)

## Session 2: Improvement Queue System

Built the Improvement Queue — centralized infrastructure for tracking and executing improvement prompts across all Sevaro repos.

### What Was Built

**AWS Infrastructure:**
- DynamoDB table `sevaro-improvement-queue` (PK: `repoName`, SK: `promptId`)
- Lambda `sevaro-improvement-queue-api` — CRUD with Cognito JWT auth, field validation, upsert protection
- API Gateway `ael0orzmsk` at `https://ael0orzmsk.execute-api.us-east-2.amazonaws.com`
- 9 items seeded from 3 repos: 5 evidence-engine, 3 hub, 1 neuroscribe

**Hub Admin Page (`/admin/improvements`):**
- Stats bar: total, pending, in-progress, completed, repo count
- Filters: repo, priority, status
- Click-to-expand cards showing full prompt text in monospace
- Copy Prompt button (clipboard), inline status dropdown, delete button
- Auth-gated via Cognito (admin only)

**Claude Code Skill (`/improvement-queue`):**
- Scans 9 repo directories for `docs/improvement-queue/*.md` files
- Lists pending prompts sorted by priority (P1 → P2 → P3)
- User selects, reads prompt + linked plan, approves execution
- After completion: updates status, suggests What's New entry

### Cross-Model Review

Codex CLI (GPT-5.3) review completed. High findings fixed:
1. Auth token missing on GET request (High — fixed)
2. PATCH upsert without ConditionExpression (High — fixed, returns 404)
3. estimatedScope not validated in PATCH (Medium — fixed)
4. No pagination on Scan (Medium — deferred, queue stays <1MB)
5. POST overwrites existing items (Medium — deferred, acceptable for admin workflow)

### Decisions Made
- Queue will stay small (<1MB), so pagination not needed yet
- POST allows overwrite — acceptable for seed + manual admin workflow
- Hardcoded fallback API URL matches existing whats-new-api pattern
