# Handoff — 2026-04-14

## Session Summary

Fixed silent 401 when admins clicked **Save Status** on a feedback session
(`/feedback/[id]`). Reported via screenshot showing
`Failed /api/feedback/28f806…?appId=evidence-engine 401` in DevTools console.

## Root Cause

Two compounding issues:

1. **Session detail page had no server-side auth gate.** The Lambda GET
   (`sevaro-feedback-api`) only requires `x-api-key`, not a JWT, so the page
   could fully render for a user whose `id_token` cookie had silently expired.
   The PATCH route (`src/app/api/feedback/[id]/route.ts`) DOES require a valid
   JWT, so the failure only surfaced on Save Status — long after the user
   thought they were "in the dashboard."

2. **`getAuthHeaders()` swallowed refresh failures.** When `getIdToken()`
   returned `null` (id_token expired *and* refresh failed), the helper
   returned headers without an `Authorization` entry instead of throwing. The
   PATCH then went out with no Bearer token, hit the cookie fallback in
   `extractToken`, verified the still-expired cookie, and returned 401 with no
   user-visible message.

## What Changed

### `src/app/feedback/[id]/page.tsx`
- Added server-side auth gate before any data fetching. Reads `id_token`
  cookie via `next/headers#cookies()`, calls `verifyToken()`. If invalid →
  `redirect('/api/auth/login')`. If valid but not admin → render
  "Admin access required" message.
- This prevents the page from ever rendering session content (or hitting
  the upstream Lambda) for an unauthenticated/expired session, eliminating
  the "stale tab" failure mode.

### `src/app/feedback/[id]/SessionDetailClient.tsx`
- Removed the silently-failing `getAuthHeaders()` helper.
- `handleStatusUpdate` and `handleDelete` now check the token explicitly:
  - If `getIdToken()` returns `null` → show "Your session has expired.
    Redirecting to sign in…" and bounce to `/api/auth/login` after 1.5s.
  - If the PATCH/DELETE returns `401` → same recovery flow.
- New `handleAuthExpired` callback centralises the messaging + redirect.

## Verification

- `npx tsc --noEmit` → exit 0
- `next dev` boots cleanly, no compile errors on the changed files
- Unauthenticated request to `/feedback/<id>?appId=…`:
  - Server returns RSC payload with
    `digest: "NEXT_REDIRECT;replace;/api/auth/login;307;"` ✅
  - **No upstream Lambda call** is made (verified via dev server network log)
  - Browser ends up on `/login` (via combined server gate + existing
    `FeedbackLayout` client redirect)
- The expired-cookie path (where the user previously hit the silent 401)
  cannot be reproduced locally without manipulating Cognito state, but the
  failure mode is structurally eliminated by the page-level
  `verifyToken` gate.

## Next Steps

- Watch for the same silent-failure pattern on `/feedback/analyze` and
  `/feedback/page.tsx`, which use the same `getIdToken()` pattern. Both
  currently throw a visible error from the loader, so they're less broken
  than the detail page was — but worth a follow-up sweep.
- Consider whether to drop the cookie fallback in `lib/verify-auth.ts`
  `extractToken()`. Removing it would force every authenticated API call to
  send an explicit `Authorization` header, removing one source of
  hard-to-diagnose 401s. Out of scope for this fix.
